# Zed 的功能开发流程

本文适用于中等规模及以上的功能开发——新的 UI、行为变更或涉及 Zed 多个部分的工作。小型的快捷键或设置调整不需要所有这些流程。

> **开始之前：**如果您是外部贡献者，请确保团队确实希望实现此功能，再投入大量精力。请阅读我们的 [功能请求指南](https://github.com/zed-industries/zed/discussions/51422) — 如果还没有一个具有明确团队确认的 GitHub 问题，请先从 GitHub 讨论开始。跳过此流程的功能请求 PR 的合并率非常低。花时间遵循我们的流程可以大大提高您的想法被采纳和实现的可能性。

## 1. 为什么这很重要？

每个功能都始于一个想法。在编写任何代码之前，请先奠定基础：

- **这解决了什么问题？**
- **有什么证据？** GitHub 问题、Discord 请求、点赞数、博客文章。
- **是否有先例？** 如果它在 VS Code、JetBrains、Neovim 或广受欢迎的插件中存在，这是一个强烈的信号。如果这个想法比较新颖，请说明它基于什么 — "这是 X，适应了 Zed 的多缓冲区功能" 比 "我觉得这会很酷" 有用得多。

## 2. 这是什么？

写一个简短、具体的功能描述，然后用上述收集到的背景信息来支撑它。如果你不能用几句话描述这个功能，它可能太庞大或太模糊。

这是一个示例格式，但请根据您的功能需求进行调整：

**功能：** 内联 Git Blame

**目的：** 在编辑器文本后直接显示每一行的最后提交作者和消息，这样开发者无需打开 git blame 就能理解代码历史。

**背景：**
这是所有主流代码编辑器的标准功能：

- [VSCode 截图]
- [Intellij 截图]
- [Neovim 截图]
- 并且在这个 [github issue](https://github.com) 上有 146 个赞。

**决策：**
我们必须决定使用 git CLI 还是 git 库。Zed 使用 git 库，但其 blame 实现对于代码编辑器来说太慢了，所以我们应该使用 CLI 的 porcelain 接口。

## 3. 这还会影响什么？

在开始构建之前，请过一遍这个列表。不是所有内容都适用：

- **操作和快捷键。** 您的功能定义了哪些操作？默认的快捷键是否与现有的快捷键冲突？
- **设置。** 任何行为是否可配置？按用户 vs 按项目 vs 按语言？别忘了在设置 UI 中添加新设置。
- **主题和样式。** 是否需要新的语义 token？在浅色和深色模式下看起来是否正确？
- **Vim 模式。** Vim 用户可能对这个功能有不同的期望。
- **远程开发。** 您的功能是否适用于远程项目？文件路径、shell 命令和环境变量可能会有不同的行为。
- **跨重启的持久化。** 您功能的状态是否应在重启后保持？
- **可访问性。** 是否可以通过键盘导航？焦点状态是否清晰？
- **平台差异。** 在 macOS、Linux 或 Windows 上行为是否有差异？
- **性能。** 处理大文件或大项目时表现如何？交互是否即时？
- **安全性。** 此功能如何与工作区信任交互？是否在 Zed 中开辟了新的攻击面？

如果您的功能特别涉及**编辑器**：编辑器有很多共存的功能 — 边栏元素、内联块、多光标、折叠、编辑预测、代码智能弹出窗口、小地图。在不同组合激活的情况下测试您的更改。在普通缓冲区中工作的功能在多缓冲区中可能需要被禁用。

## 4. 发布

将此作为您的 GitHub 讨论、问题或 PR 描述的基础。良好的产品研究可以让每个人就目标、技术现状以及任何我们需要考虑的权衡达成一致。
